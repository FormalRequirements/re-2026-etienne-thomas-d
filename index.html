2. Goals

2.1. G-01: Objectif global

Ref PEGS: G.1

Priorité: Très haute

Description
Permettre à un pratiquant de suivre ses entraînements de musculation/fitness simplement, de façon fiable et rapide.

Justification

Alignement produit : app utile au quotidien.

Critères d’acceptation
Un utilisateur peut enregistrer une séance complète en moins de 3 minutes de saisie cumulée (hors effort).

2.2. G-02: Situation actuelle

Ref PEGS: G.2

Priorité: Très haute

Description
Le pratiquant utilise actuellement notes papier/app générique, avec perte de données et manque de structure.

Justification

Justifie la valeur : structure + historique + progression.

Critères d’acceptation
Le document PEGS décrit au moins 2 solutions actuelles et leurs limites.

2.3. G-03: Bénéfices attendus

Ref PEGS: G.3

Priorité: Très haute

Description
L’app doit aider à : (1) gagner du temps à la saisie, (2) visualiser la progression, (3) favoriser la régularité.

Justification

Valeur utilisateur immédiate.

Critères d’acceptation
Des écrans dédiés existent pour : saisie rapide + progression + fréquence.

2.4. G-04: Scénarios usage (mobile-first)

Ref PEGS: G.5

Priorité: Haute

Description
L’app doit être utilisable d’une main en salle : grands boutons, parcours court, interactions minimales.

Justification

Usage réel en entraînement, contraintes de contexte.

Critères d’acceptation
Un parcours 'Démarrer séance → saisir série → terminer' ≤ 5 actions principales.

2.5. G-05: Exclusions V1

Ref PEGS: G.6

Priorité: Haute

Description
La V1 exclut : marketplace coach, paiements, messagerie temps réel, IA d’analyse avancée, intégrations montres/santé.

Justification

Évite les pièges à temps ; garde le focus sur le core.

Critères d’acceptation
La liste des exclusions est visible et validée.

2.6. G-06: Parties prenantes

Ref PEGS: G.7

Priorité: Très haute

Description
Les parties prenantes minimum sont : pratiquant (utilisateur principal), développeur (solo), évaluateur (prof).

Justification

Clarifie les attentes et l’arbitrage.

Critères d’acceptation
Une section 'Stakeholders' liste besoins et critères de succès.

3. Environment

3.1. E-01: Contraintes d’usage en salle

Ref PEGS: E.3

Priorité: Très haute

Description
L’app doit fonctionner avec réseau instable et permettre la saisie sans dépendre d’une connexion continue.

Justification

En salle : 4G/Wi-Fi souvent irréguliers.

Critères d’acceptation
Mode offline : créer une séance et saisir des séries sans erreur même sans réseau.

3.2. E-02: Contraintes de temps

Ref PEGS: E.3

Priorité: Très haute

Description
La saisie d’une série doit pouvoir se faire en < 5 secondes via UI optimisée (valeurs récentes, boutons +/-, auto-focus).

Justification

Saisie rapide = adoption.

Critères d’acceptation
Test manuel : saisie d’une série en < 5s sur mobile milieu de gamme.

3.3. E-03: Hypothèse matériel

Ref PEGS: E.4

Priorité: Haute

Description
L’utilisateur dispose d’un smartphone Android/iOS récent (≤ 5 ans) et d’un navigateur moderne.

Justification

Cadrage réaliste sans viser les vieux appareils.

Critères d’acceptation
Liste des environnements testés (au moins 1 Android + 1 navigateur desktop).

3.4. E-04: Invariant d’intégrité

Ref PEGS: E.6

Priorité: Très haute

Description
Une série validée ne doit jamais disparaître sans action explicite de l’utilisateur (suppression/annulation).

Justification

La perte de données tue la confiance.

Critères d’acceptation
Après redémarrage app, les séries saisies sont toujours présentes.

3.5. E-05: Contraintes confidentialité (RGPD)

Ref PEGS: E.3

Priorité: Très haute

Description
L’app doit minimiser les données personnelles (principe de minimisation) et permettre suppression/export des données utilisateur.

Justification

Exigence standard et attendue sur une app fitness.

Critères d’acceptation
L’utilisateur peut exporter et supprimer ses données depuis les réglages.

3.6. E-06: Glossaire

Ref PEGS: E.1

Priorité: Moyen

Description
Un glossaire doit définir les termes clés : séance, exercice, série, répétition, charge, PR, template, programme.

Justification

Réduit les incompréhensions dans les exigences et l’UI.

Critères d’acceptation
Le glossaire contient au moins 10 termes et est utilisé dans les descriptions.

4. System

4.1. S-01: Composants - Mobile/Web

Ref PEGS: S.1

Priorité: Très haute

Description
Le système doit inclure au minimum : une interface mobile (PWA ou app), une interface web desktop, et une persistance de données fiable (local + serveur).

Justification

Vision 'app globale' (pas juste API).

Critères d’acceptation
Un schéma des composants est fourni.

4.2. S-02: Composants - Admin contenu

Ref PEGS: S.1

Priorité: Haute

Description
Le système doit prévoir un espace d’administration pour gérer la bibliothèque d’exercices (CRUD, tags, groupes musculaires).

Justification

Évite d’éditer en base / facilite l’évolution contenu.

Critères d’acceptation
Un admin peut ajouter/modifier un exercice et le voir côté utilisateur.

4.3. S-10: Compte utilisateur

Ref PEGS: S.2

Priorité: Très haute

Description
L’app doit permettre la création de compte, la connexion, la déconnexion et la récupération de mot de passe.

Justification

Base d’un usage multi-appareils + sauvegarde.

Critères d’acceptation
Création/connexion/déconnexion testées avec succès.

4.4. S-11: Profil & préférences

Ref PEGS: S.2

Priorité: Haute

Description
L’utilisateur doit pouvoir définir ses préférences : unités (kg/lb), langue (si supportée), thème (clair/sombre), repos par défaut.

Justification

Personnalisation minimale utile en salle.

Critères d’acceptation
Les préférences impactent l’affichage (ex : kg/lb) immédiatement.

4.5. S-12: Bibliothèque d’exercices

Ref PEGS: S.2

Priorité: Très haute

Description
L’utilisateur doit pouvoir consulter et rechercher des exercices, filtrer par groupe musculaire, et voir une fiche (consignes, muscles ciblés).

Justification

Accélère la saisie et réduit les erreurs.

Critères d’acceptation
Recherche + filtre fonctionnels ; fiche exercice affichée.

4.6. S-13: Création de séance

Ref PEGS: S.2

Priorité: Très haute

Description
L’utilisateur doit pouvoir créer une séance avec nom, date, et liste d’exercices.

Justification

Noyau fonctionnel.

Critères d’acceptation
Créer une séance avec ≥ 3 exercices et la retrouver dans l’historique.

4.7. S-14: Exécution de séance (tracking)

Ref PEGS: S.2

Priorité: Très haute

Description
L’utilisateur doit pouvoir démarrer une séance et enregistrer, pour chaque exercice, des séries (répétitions, charge, RPE optionnel).

Justification

Suivi réel en salle.

Critères d’acceptation
Enregistrer ≥ 10 séries sur une séance et terminer la séance.

4.8. S-15: Minuteur de repos

Ref PEGS: S.2

Priorité: Haute

Description
L’app doit proposer un minuteur de repos configurable par exercice ou global, déclenchable en un geste après validation de série.

Justification

Soutient le flux d’entraînement.

Critères d’acceptation
Après validation série, minuteur démarre en ≤ 1 action.

4.9. S-16: Saisie rapide (valeurs précédentes)

Ref PEGS: S.2

Priorité: Haute

Description
Lors d’une série, l’app doit préremplir les champs avec la dernière série réalisée sur cet exercice (ou séance précédente).

Justification

Gros gain de temps.

Critères d’acceptation
Préremplissage visible et modifiable ; testé sur 2 séances successives.

4.10. S-17: Historique des séances

Ref PEGS: S.2

Priorité: Très haute

Description
L’utilisateur doit pouvoir consulter l’historique des séances (liste + détail) et filtrer par période.

Justification

Accès aux données enregistrées.

Critères d’acceptation
Afficher les séances des 30 derniers jours + détail d’une séance.

4.11. S-18: Records (PR)

Ref PEGS: S.2

Priorité: Moyen

Description
L’app doit détecter et afficher les records personnels (charge max, reps max à charge donnée, volume) par exercice.

Justification

Motivation et lecture progression.

Critères d’acceptation
Un PR est mis à jour automatiquement après une nouvelle perf.

4.12. S-19: Statistiques simples

Ref PEGS: S.2

Priorité: Moyen

Description
L’app doit afficher des statistiques simples : volume hebdo, fréquence, top exercices, temps moyen de séance.

Justification

Bénéfice attendu G.3.

Critères d’acceptation
Un écran stats existe et reflète les séances saisies.

4.13. S-20: Templates / séances types

Ref PEGS: S.2

Priorité: Haute

Description
L’utilisateur doit pouvoir sauvegarder une séance comme template et relancer un entraînement à partir d’un template.

Justification

Accélère l’usage récurrent.

Critères d’acceptation
Créer un template et démarrer une séance depuis ce template.

4.14. S-21: Modification/annulation

Ref PEGS: S.2

Priorité: Haute

Description
L’utilisateur doit pouvoir modifier une série (corriger une charge/reps) et annuler la dernière action rapidement.

Justification

Évite la frustration en saisie mobile.

Critères d’acceptation
Modifier une série existante et vérifier recalcul des stats.

4.15. S-22: Sauvegarde & synchronisation

Ref PEGS: S.2

Priorité: Très haute

Description
Les données doivent être sauvegardées localement immédiatement, puis synchronisées au serveur dès que possible.

Justification

Fiabilité + offline.

Critères d’acceptation
Couper le réseau : saisie ok ; remettre réseau : synchro se fait sans perte.

4.16. S-23: Gestion conflits simple

Ref PEGS: S.2

Priorité: Moyen

Description
En cas de modification concurrente (2 appareils), le système doit appliquer une stratégie simple : dernier enregistrement gagne, avec avertissement.

Justification

Évite une usine à gaz tout en restant cohérent.

Critères d’acceptation
Simuler 2 modifications : l’utilisateur voit un message de conflit et le résultat final.

4.17. S-24: Export données

Ref PEGS: S.2

Priorité: Moyen

Description
L’utilisateur doit pouvoir exporter ses données (CSV/JSON) depuis les réglages.

Justification

Portabilité + devoir + RGPD.

Critères d’acceptation
Téléchargement export ; fichier contient séances + séries.

4.18. S-25: Suivi mensurations

Ref PEGS: S.2

Priorité: Faible

Description
L’app peut permettre de saisir poids corporel et mensurations (optionnel).

Justification

Valeur mais non core V1.

Critères d’acceptation
Ajouter une mesure et l’afficher sur une courbe simple.

4.19. S-26: Partage séance

Ref PEGS: S.2

Priorité: Faible

Description
L’app peut permettre de partager un résumé de séance (image ou lien).

Justification

Nice-to-have ; risque de dérive.

Critères d’acceptation
Partager un résumé lisible via applications du téléphone.

4.20. S-30: Interface utilisateur responsive

Ref PEGS: S.3

Priorité: Très haute

Description
L’interface doit être responsive et optimisée mobile (priorité), tout en restant utilisable sur desktop.

Justification

App globale : mobile + web.

Critères d’acceptation
Même parcours possible sur mobile et desktop sans blocage.

4.21. S-31: Accessibilité de base

Ref PEGS: S.3

Priorité: Moyen

Description
L’app doit respecter une accessibilité minimale : taille de police lisible, contrastes corrects, navigation clavier sur web.

Justification

Qualité et inclusivité sans surcoût énorme.

Critères d’acceptation
Audit manuel : zoom 200% ok ; tabulation web ok.

4.22. S-32: Mode sombre

Ref PEGS: S.3

Priorité: Moyen

Description
L’app doit proposer un mode sombre (au moins sur mobile).

Justification

Confort en salle/faible lumière.

Critères d’acceptation
Toggle mode sombre ; persistance du choix.

4.23. S-40: Scénario: Démarrer séance

Ref PEGS: S.4

Priorité: Très haute

Description
Scénario : depuis l’accueil, l’utilisateur démarre une séance (template ou nouvelle), voit la liste d’exercices et peut saisir la première série.

Justification

Valide le flux principal.

Critères d’acceptation
Parcours testé de bout en bout sans navigation inutile.

4.24. S-41: Scénario: Offline en salle

Ref PEGS: S.4

Priorité: Très haute

Description
Scénario : réseau coupé, l’utilisateur exécute la séance complète ; à la reconnexion, la séance est synchronisée.

Justification

Critique pour la confiance.

Critères d’acceptation
Test en mode avion : séance complète sauvegardée puis sync.

4.25. S-42: Scénario: Consultation progression

Ref PEGS: S.4

Priorité: Moyen

Description
Scénario : l’utilisateur ouvre un exercice et voit l’évolution (dernières perfs + PR).

Justification

Bénéfice progression.

Critères d’acceptation
Vue 'historique exercice' affiche au moins 5 dernières performances.

4.26. S-50: Priorisation V1

Ref PEGS: S.5

Priorité: Très haute

Description
La V1 doit prioriser : tracking séance, bibliothèque exercices, historique, sauvegarde/sync. Tout le reste est différable.

Justification

Time-to-code : éviter l’over-engineering.

Critères d’acceptation
Backlog montre ces items en MUST ; V2/V3 en SHOULD/COULD.

4.27. S-51: Performance perçue

Ref PEGS: S.5

Priorité: Haute

Description
Les écrans principaux doivent s’ouvrir en < 1 seconde sur un smartphone milieu de gamme (hors 1er chargement).

Justification

Adoption : la lenteur tue l’usage.

Critères d’acceptation
Mesure manuelle : navigation accueil→séance < 1s.

4.28. S-60: Critères d’acceptation testables

Ref PEGS: S.6

Priorité: Très haute

Description
Chaque fonctionnalité V1 doit être validée par un scénario de test manuel reproductible (Given/When/Then).

Justification

Aide devoir + qualité sans tests lourds.

Critères d’acceptation
Un fichier de tests manuels couvre 100% des MUST V1.

4.29. S-61: Fiabilité des données

Ref PEGS: S.6

Priorité: Très haute

Description
Après crash/restart, l’app doit retrouver l’état stable (séances et séries sauvegardées), sans corruption.

Justification

Invariant E.6 appliqué.

Critères d’acceptation
Test : forcer arrêt app puis relance : données intactes.

4.30. S-62: Sécurité minimale

Ref PEGS: S.6

Priorité: Haute

Description
Les données en transit doivent être protégées (HTTPS) et les mots de passe stockés de façon sécurisée (hash robuste).

Justification

Hygiène de base, sans faire une thèse sécurité.

Critères d’acceptation
Scan : pas de mot de passe en clair ; connexion via HTTPS.

5. Project

5.1. P-01: Rôles & responsabilités

Ref PEGS: P.1

Priorité: Très haute

Description
Le projet doit définir les rôles minimum suivants : Pratiquant (utilisateur), Administrateur (gestion contenu) et, si activé, Coach (accès encadré).

Justification

Évite les ambiguïtés et oriente la conception d’écrans/droits.

Critères d’acceptation
Une matrice rôles→droits est disponible et couvre toutes les fonctionnalités V1.

5.2. P-02: Livrables V1

Ref PEGS: P.4

Priorité: Très haute

Description
La V1 doit livrer une application utilisable en conditions réelles pour enregistrer des séances (création, exécution, historique) avec sauvegarde fiable.

Justification

Dogfooding rapide : valider l’usage avant d’ajouter des features.

Critères d’acceptation
Un utilisateur peut enregistrer 10 séances complètes sur mobile sans bug bloquant.

5.3. P-03: Découpage versions

Ref PEGS: P.3

Priorité: Haute

Description
Le projet doit maintenir un découpage V1/V2/V3 avec une liste explicite des exclusions de V1.

Justification

Protège le Time-to-Code et évite de s’éparpiller

Critères d’acceptation
Un document 'Scope V1' existe, avec une section 'Non inclus'.

5.4. P-04: Process exigences

Ref PEGS: P.7

Priorité: Très haute

Description
Les exigences doivent être tracées (id unique), classées PEGS, et associées à des critères d’acceptation mesurables.

Justification

Facilite l’évaluation académique et le pilotage.

Critères d’acceptation
100% des exigences ont id + PEGS + MoSCoW + critères d’acceptation.

5.5. P-05: Technologie cible (app)

Ref PEGS: P.5

Priorité: Haute

Description
L’application doit être utilisable sur mobile en priorité et accessible sur navigateur (desktop) pour consultation et gestion.

Justification

Maximise l’usage réel sans multiplier les apps natives au départ.

Critères d’acceptation
La même base fonctionnelle est accessible sur téléphone et desktop.

5.6. P-06: Gestion risques

Ref PEGS: P.6

Priorité: Haute

Description
Le projet doit identifier 5 risques majeurs (données perdues, offline, perf mobile, complexité UX, dette) et des mitigations simples.

Justification

Réduit les échecs classiques d’une app perso en solo-dev.

Critères d’acceptation
Un tableau risques/impact/mitigation existe.
